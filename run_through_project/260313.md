### SOP
- SOP : Same Origin Policy
- 어떤 출처에서 불러온 문서나 스크립트가 다른 출처에서 가져온 리소스와 상호작용하는 것을 제한하는 보안 방식
- -> 다른 곳에서 가져온 자료는 일단 막는다
- -> 웹 애플리케이션 도메인이 다른 도메인의 리소스에 접근하는 것을 제어하여 사용자의 개인 정보와 데이터 보안 보호, 잠재적 보안 위협 방지
- -> 잠재적으로 해로울 수 있는 문서 분리해 공격받을 수 있는 경로 줄임
- 같은 로컬호스트라도 포트 번호가 다르면 다른 출처로 간주. 백은 8000 프론트는 5173
- -> CORS 규약에 의해 막혔다는 에러 메시지 발생
- 장고에 settings.py에 설정 가능.
  - CORS_ALLOWED_ORIGINS : 허용할 출처 목록('http://127.0.0.1:5173' 적어주기)
- url의 protocol, domain, port가 모두 일치해야 같은 출처로 간주
- 퍼블릭한 서비스를 만들 때 https로 처리해주는 정도만으로도 많은 보안 문제 해결 가능 
### CORS policy 등장
- CORS : Cross-Origin Resource Sharing(교차 출처 자원 공유)
- 기본적으로 웹 브라우저는 같은 출처에서 요청하는 것만 허용, 다른 출처로의 요청은 보안상 이요로 차단
- 현대 웹 애플리케이션은 다양한 출처로부터 리소스를 요청하는 경우 많음 -> CORS 정책 등장
- 다른 출처의 리소스를 불러오려면 그 다른 출처에서 올바른 CORS 헤더를 포함한 응답을 보내야 함
- 서버에서 설정됨, 브라우저가 해당 정책 확인해 요청 허용 여부 결정
### CORS Headers 설정하기
- 장고에서는 django-cors-headers 패키지 사용하여 CORS 헤더 설정 가능
- 설치 후 settings.py에 'corsheaders' 추가, CORS_ALLOWED_ORIGINS 설정하여 허용할 출처 지정
- middleware에 'corsheaders.middleware.CorsMiddleware' 추가하여 CORS 헤더가 모든 응답에 포함되도록 설정
- corsheaders.middleware.CorsMiddleware는 Django의 CommonMiddleware보다 먼저 추가되어야 함
- 주의
  - 장고는 vscode에서 창 여러개 열어서 실행하면 전부 포트번호 8000으로 실행됨 -> 첫번째로 연 서버로만 요청이 감
  - 뷰는 여러 개 켜면 포트번호 1씩 늘어남. -> 발표 여러 팀이 하면서 장고 서버 안끄고가면 요청 이상하게 가서 난리난다.. 발표 끝나고 vscode 창 다 꺼버리기
  - 발표 끝나고 서버꺼라
- CORS 설정은 개발 단계에서만 허용하는 것이 좋음. 배포 시에는 보안상의 이유로 필요한 출처만 허용하도록 설정하는 것이 좋음
### 기획 설계의 중요성
- 일주일 코드쓴다 생각하면 앞에 2주정도는 기획/설계 단계에 투자하는 게 좋음
- 기획 단계에서 API 명세, 데이터 모델링, 프론트엔드와 백엔드 간의 인터페이스 정의 등 프로젝트의 전체 구조와 흐름을 명확히 하는 것이 중요
- 기획이 잘 되어 있으면 개발 과정에서 발생할 수 있는 혼란과 오류을 줄이고, 개발 속도를 높일 수 있음

- 기획 단계때 목업 다 짜둔대로 기본 로직들 짜두고 더미데이터로 목업 레이아웃 대로 화면 완성해두기.
