# Generater
# 메모리 최적화
## 개요
### List(Eager Evaluation - 즉시 평가)
  - 리스트는 PyObject* 포인터들을 담는 동적 배열
  - 원소 객체를 가리키는 포인터 배열 연속으로 저장하는 것
  - 요소 N개를 담으려면 포인터 슬롯 N개 이상 필요
  - 원소가 객체라면 객체 메모리도 별도로 필요
  - -> 큰 데이터를 리스트로 한 번에 만들면 메모리 사용량 증가 -> OOM 위험 올라감
### 제너레이터(Generator)
- 필요할 때마다 값을 하나씩 생성해 반환하는 효율적인 반복 객체
### list vs generator
- tracemalloc은 파이썬에서 메모리 할당 추적하는 디버깅용 모듈
- 코드가 어디서 얼마나 메모리 할당했는지를 파일, 라인 단위로 추적
- 특정 시점의 메모리 상황을 스냅샷으로 찍고, 스냅샷을 비교하여 어디서 얼마나 더 늘어났는지 확인
- get_traced_memory()로 메모리 사용량 체크 가능
- generator는 데이터를 한 번만 흘려 보내며 처리할 때 메모리 효율이 좋음
- 결과를 여러 번 반복해서 사용하거나 인덱싱이 필요하면 list로 물질화하는 것이 이득일 수 있음
## iterator
- 반복 가능한 객체의 요소를 하나씩 반환하는 객체
### 파이썬 내부 반복 동작 원리
- Iterable : iter(x)가 가능한 객체(반복 시작 가능)
- Iterator : next(it)가 가능한 객체(다음 값을 하나씩 내놓음)
- for문의 내부 동작: it=iter(x)로 iterator를 얻고 next(it) 반복하다가 StopIteration 만나면 종료
### Iterator 객체 구현
- __iter__() 메서드로 iterator 객체 반환
- __next__() 메서드로 다음 요소 반환, 더 이상 요소 없으면 StopIteration 예외 발생
- iterator는 보통 __iter__()에서 self 반환하여 자신이 iterator임을 나타냄
- for는 iter()로 시작하고 next()로 진행한다는 규칙을 만족시키는지 확인
### iterator의 단점
- 상태를 객체 필드로 직접 관리 필요(초기화/증가/종료 조건)
- StopIteration 예외 처리, __iter__/__next__ 보일러플레이트 필요
- 한번 소비된 iterator는 기본적으로 재사용 어려움
- **간단한 스트리밍 로직을 만들기엔 너무 장황함** -> yield로 넘어감
- yield는 함수의 실행을 완전히 끝내지 않고 값을 반환한 뒤 그 자리에 일시 정지시켜 필요할 때마다 데이터를 하나씩 꺼내 쓸 수 있는 제너레이터 만드는 키워드
## generator
- 반복 가능한 객체의 요소를 하나씩 반환하는 객체(iterator)를 간단하게 만드는 객체
### 제너레이터 함수 구조
- 일반적 함수처럼 작성
- yield문을 사용하여 결과값이 아니라 값을 차례대로 꺼낼 수 있는 제너레이터 객체 반환 
### yield의 핵심
- 제너레이터 함수는 호출 시 바로 실행되는 게 아닌 제너레이터 객체를 반환
- next()(for())가 호출될 대마다 함수가 이전 위치부터 재개(resume)
- yield는 두 가지 담당
  1. 값을 바깥으로 내보냄
  2. 현재 실행 지점/로컬 변수 상태 저장
- => 이러한 특성으로 제너레이터를 1회성 처리에 강함
### 제너레이터 특징
- 클래스 기반 이터레이터 만들 필요 없이 __iter__과 __next__메서드 저절로 만들어짐
- self.index나 self.data 같은 인스턴스 변수 사용하는 접근법에 비해 쓰기 쉽고 명료함 
### 제너레이터 사용 이유
1. 메모리 효율성
  - 한 번에 한 개의 값만 생성 : 제너레이터는 값 하나씩 생성하여 반환, 전체 시퀀스 한번에 메모리에 로드하지 않아도 됨
  - 대용량 데이터 처리 : 대용량 데이터셋 처리 시 메모리 사용 최소화 가능. 예, 파일의 한 줄을 하나씩 읽어 처리
2. 무한 시퀀스 처리
  - 무한 시퀀스 생성 : 무한 루프를 통해 무한 시퀀스 생성 가능. 끝 없는 데이터 스트림 처리 시 유용
3. 지연 평가(Lazy Evaluation)
  - 필요할 때만 값 생성 : 제너레이터는 값이 실제로 필요할 때만 값 생성, 불필요한 계산 피하고 성능 최적화 가능
  - 연산 지연 : 복잡한 연산 지연 수행 가능, 계산이 필요한 시점에만 연산 이루어짐
## 제너레이터 활용
1. 무한 시퀀스 생성
  - 필요할 때마다 값 생성하여 무한 반복 (피보나치 수열 등)
2. 대용량 데이터 처리
  - 전체 데이터 한 번에 메모리에 로드하지 않고 필요한 시점에 생성
  - 파일에서 한 줄씩 읽어와 처리
3. yield from
  - yield from : 파이프라인 중계
  - 다른 iterable/generator의 값을 전달만 하고 싶을 때 사용
  - 반복+yield를 한 줄로 표현하는 문법
    - 메모리에 전체 파일을 로드하지 않고 한 줄씩 처리하여 메모리 사용 절약
    - 동작 자체는 other_iterable을 순회하며 yield하는 것과 동일
### 제너레이터와 데이터 분석에서의 활용
- 추후 데이터 분석 시 큰 데이터를 한꺼번에 받아서(메모리에 올려서) 처리할 수 없는 경우 발생
- 데이터를 개발자가 원하는 양만큼만 가져와서 처리, 다시 그 양만큼만 가져와서 처리하는 행위 반복하여 데이터 분석 진행
- generator/iterator가 유리한 경우
  - 입력이 크고 1회성 처리가 목적
  - 중간 결과 저장 없이 파이프라인으로 연결
  - 앞 단계에서 일부만 소비(early stopping)하는 경우
- list(물질화)가 유리한 경우
  - 결과 여러 번 반복해서 사용, 인덱싱/길이/랜덤 접근 필요
  - 정렬/슬라이싱 등 전체를 알아야 하는 연산 필요
  - 계산을 한 번 해 두고 캐시처럼 재사용
### 참고
- inspect : 객체의 정보 얻을 수 있는 함수 제공 모듈
- 제너레이터 표현식
  - 리스트 컴프리헨션과 비슷하지만 대괄호 대신 소괄호 사용
  - 일반적으로 리스트 컴프리헨션보다 메모리 덜 사용
  - 간결하지만 가독성 떨어짐
- 제너레이터 주의사항
  - 제너레이터 요소 모두 참조한 경우 재사용 불가
  - 다시 제너레이터 함수 사용, 변수에 재할당 필요

# 비동기 동작
## 개요
### 동기 코드의 한계 - 블로킹
- 동기(synchronous) 코드 : 명령이 순차적으로 실행되는 코드. 작업 1이 끝날 때까지 다음 작업 2로 못 넘어감
- I/O(네트웥크/디스크) 대기 시간은 CPU를 놀게 만든다(대기=블로킹)
  - I/O 작업을 여러 개 겹쳐서 처리하면 체감 성능 개선
## 동시성
- 여러 작업을 겹쳐 진행하는 구조
### 동시성
- 반드시 동시에 실행되는 것은 아님
  - 대기 시간을 다른 작업으로 때운다
- I/O 중심 프로그램에서는 CPU보다 기다림이 병목이 되는 경우 많음
  - 목표는 CPU를 더 쓰는 것이 아니라 기다리는 시간을 줄이는 것
### 병렬성과의 차이
- 병렬성은(parallelism) 실제로 여러 코어에서 동시에 실행되는 것
- 동시성은 한 코어에서도 가능
- I/O 바운드 작업 -> 동시성이 효과적
  - 데이터 입력과 출력(네트워크 요청, 파일 읽기/쓰기, 데이터베이스 쿼리 등)
  - 기다리는 시간이 실제 계산 시간보다 많은 작업
- CPU 바운드 작업 -> 병렬성이 효과적
  - 복잡한 수학 연산, 이미지 처리, 암호화 등 CPU가 쉴 새 없이 계산해야 하는 작업
- 이 구분이 threading / multiprocessing / asyncio 선택 기준이 됨
### 프로세스 / 스레드
- 프로세스 : 독립된 메모리 공간을 가진 실행 단위
- 스레드 : 같은 프로세스 내에서 메모리 공유하는 실행 단위
### threading의 한계
- 파이썬(CPython)에는 GIL(Global Interpreter Lock)이 있음
  - 멀티 코어 환경에서도 멀티 스레드가 실제로 동시에 연산하지 못하게 막는 잠금 방식
  - 파이썬의 메모리 관리 방식(Reference Counting)을 안전하게 보호하기 위해 존재
    - Reference Counting : 모든 객체가 자신을 참조하는 변수나 데이터의 개수를 직접 기억하고 관리하는 방식
- 결과
  - I/O 바운드 + threading -> 효과 있음
  - CPU 바운드 + threading -> 제한적 (GIL 때문에)
### 파이썬 동시성
- threading (I/O 대기가 많은가?)
  - 장점: 구현 간단, I/O바운드 적합
  - 단점: GIL로 CPU 바운드에 비효율적
- multiprocessing (CPU 사용량이 많은가?)
  - 장점: 프로세스마다 독립, GIL -> CPU 병렬화 가능
  - 단점: 프로세스 생성 비용, 메모리 사용 증가
- asyncio (동시 연결 수가 많은가?)
  - 장점: 싱글 스레드 이벤트 루프 기반, 대량 I/O 처리에 효율적
  - 단점: 코드 구조 변화 필요, CPU 작업에는 직접적 도움 없음
## asyncio
- 싱글 스레드 기반의 cooperative concurrency(협력적 동시성) 모델
### 이벤트 루프 모델
- 준비된 작업 실행 
- I/O 대기 중인 작업 등록
- 준비되면 다시 이어서 실행
- OS레벨에서는 select/epoll/IOCP 같은 비동기 I/O 매커니즘 사용
  - select: 관리하는 모든 소켓을 처음부터 끝까지 훑으며 데이터 왔는지 확인
  - epoll(Linux): I/O 작업이 가능한 상태(Readiness)가 되었음을 통지
  - IOCP(Windows): I/O 작업이 완료되었음을 통지
- 스레드 늘리는 대신 **한 스레드에서 많은 작업 스케줄링**하는 방식
### async def / await
- async def : 코루틴(coroutine) 함수 정의
  - 코루틴 함수: 실행을 중간에 일시 정지했다가 나중에 멈춘 지점부터 다시 시작할 수 있는 함수
- 호출 시 즉시 실행되지 않고 코루틴 객체 생성
- await : 해당 작업이 완료될 때까지 현재 코루틴 일시 중단
- 중단되는 동안 이벤트 루프가 다른 준비된 코루틴 실행
### 블로킹 코드와 asyncio 충돌
- asyncio는 협력적 모델
- 코루틴이 await를 만나야 다른 작업 실행
- 코루틴 내부에서 블로킹 함수(time.sleep, CPU 연산 등)를 호출하면 이벤트 루프 전체 멈춤
### create_task vs gather
구분 | asyncio.create_task | asyncio.gather
---|---|---
목적 | 단일 코루틴을 백그라운드 작업으로 분리 | 여러 코루틴을 동시에 실행하고 결과 모으기
반환값 | 개별 Task 객체 (나중에 await 가능) | 코루틴 결과 리스트 (실행 완료 후)
제어권 | 태스크 생성 후 바로 다음 코드로 제어권 넘어감 | 묶인 작업들이 모두 끝날 때까지 해당 줄에서 블로킹
활용 예시 | 서브 구동시켜두고 메인 루프에서 사용자 입력 받기 | 3개의 서로 다른 api 에서 동시에 데이터 가져와 취합
- 장기 실행 작업 등록 -> create_task
- 단순 병렬 실행 + 결과 수집 -> gather
### run_in_executor
- async 함수에서 동기(블로킹) 함수 써야 하는 경우
  - 오래 걸리는 파일 처리, 기존 라이브러리 함수, CPU 계산 함수 등
- 그냥 호출하면 이벤트 루프 멈춤
- 이벤트 루프는 유지하되 실제 블로킹 작업은 별도 실행자(executor)에 위임
- 완료되면 다시 이벤트 루프로 결과 반환
- loop.run_in_executor(executor, func, *args) 형태로 사용
  - executor: None(기본 ThreadPoolExecutor), ProcessPoolExecutor(CPU 병렬화 기능) 등
  - func: 실행할 동기 함수
  - args: func에 전달할 인자들
## 참고
### ProcessPoolExecutor
- ThreadPoolExecutor는 GIL의 영향을 받음
- CPU 연산을 병렬로 실행하려면 프로세스 기반 실행자 필요
- ProcessPoolExecutor는 각 작업을 별도 프로세스에서 실행
- 각 프로세스는 독립 GIl을 가지므로 실제 병렬 실행 가능
### 멀티프로세싱
- CPython에는 GIL 존재
- threading은 CPU Bound 작업에서 병렬 실행 되지 않음
- CPU 연산을 실제로 병렬 처리하려면 프로세스 분리 필요
- 각 프로세스는 독립 메모리 공간
- 각 프로세스는 독립 GIL 보유 -> 실제 병렬 실행 가능
- 단점
  - 프로세스 생성 비용
  - 메모리 사용량 증가
