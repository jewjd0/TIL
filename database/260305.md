# SQL Advanced
## JOIN
- 관계: 여러 테이블 간의 논리적 연결
- 게시물을 작성한 사용자 정보 조회할 때 작성자로만 찾으면 동명이인 존재 가능
- -> 테이블을 나눠서 분류하고 외래 키 참조하여 연결
- 테이블을 분리하면 데이터 관리는 용이하나 출력이 어려움 -> 테이블 하나만으로는 출력할 수 없음 -> 다른 테이블과 결합하여 출력 -> JOIN사용
### JOIN clause
- 둘 이상의 테이블에서 데이터를 검색하는 방법
### INNER JOIN
- 두 테이블에서 값이 일치하는 레코드에 대해서만 결과 반환
- ```SQL
    SELECT column1, column2, ...
    FROM table1
    INNER JOIN table2 
    ON table1.common_column = table2.common_column;
  ```
- FROM절 이후 메인 테이블 지정
- INNER JOIN 절 이후 조인할 테이블 지정
- ON 키워드 이후 조인 조건 작성
- 조인 조건은 table1과 table2간의 레코드를 일치시키는 규칙 지정
- 각 테이블에서 조인할 컬럼에 데이터가 없는 레코드는 결과에 포함되지 않음
- 자주 사용하는 조인 유형이라 INNER JOIN은 JOIN으로 생략해서 작성하기도 함
### LEFT JOIN
- 오른쪽 테이블의 일치하는 레코드와 함께 왼쪽 테이블의 모든 레코드 반환
- ```SQL
    SELECT column1, column2, ...
    FROM table1
    LEFT JOIN table2 
    ON table1.common_column = table2.common_column;
  ```
- FROM 절 이후 왼쪽 테이블 지정 (순서 중요)
- LEFT JOIN 절 이후 오른쪽 테이블 지정
- ON 키워드 이후 조인 조건 작성
  - 왼쪽 테이블의 각 레코드를 오른쪽 테이블의 모든 레코드와 일치시킴
- 오른쪽 테이블에 일치하는 레코드가 없는 경우 NULL로 표시(왼쪽 테이블 레코드는 무조건 다 포함)
- 활용: 게시글을 작성한 이력이 없는 회원 조회. user LEFT JOIN article해서 article의 id가 NULL인 경우 -> 게시글 작성한 이력이 없는 회원
- 왼쪽 테이블에 있는 걸 무슨 일이 있어도 다 써야 하는데 오른쪽 테이블에 있는 걸 참조하면서 다 쓸 필요는 없는 경우 LEFT JOIN 사용
### RIGHT JOIN
- 왼쪽 테이블의 일치하는 레코드와 함께 오른쪽 테이블의 모든 레코드 반환
- ```SQL
    SELECT column1, column2, ...
    FROM table1
    RIGHT JOIN table2 
    ON table1.common_column = table2.common_column;
  ```
- FROM 절 이후 왼쪽 테이블 지정 (순서 중요)
- RIGHT JOIN 절 이후 오른쪽 테이블 지정
- ON 키워드 이후 조인 조건 작성
  - 오른쪽 테이블의 각 레코드를 왼쪽 테이블의 모든 레코드와 일치시킴
- 왼쪽 테이블에 일치하는 레코드가 없는 경우 NULL로 표시(오른쪽 테이블 레코드는 무조건 다 포함)
- left join과 right join은 순서에 따라 결과가 동일하게 나올 수 있어 잘 사용하지 않음. 코드 가독성이 헷갈릴 때 right join 사용하기도 함
  - 복잡하고 연속된 조인이 필요할 때 right join이 left join보다 가독성 좋을 때가 있음
### SELF JOIN
- 동일한 테이블의 컬럼을 비교하여 일치하는 데이터를 추가로 붙여 반환
- 주로 계층적 데이터 구조 표현, 동일 테이블 내에서 특정 관계 찾을 때 사용
- ```SQL
    SELECT a.column1, b.column2, ...
    FROM table_name a
    JOIN table_name b 
    ON a.common_column = b.common_column;
  ```
- FROM 절 이후 본 테이블 지정 (별칭 사용 권장)
- JOIN 절 이후 본 테이블 지정 (별칭 사용 필수)
- ON 키워드 이후 조인 조건 작성
  - 동일 테이블의 컬럼을 비교하여 일치하는 레코드 찾음
- 부모의 id와 자식의 parent_id가 일치하는 경우 부모-자식 관계 표현 가능
- 개념적으로 self join이고 결국 inner join이라 생각하면 됨. 

## Subquery
- 하나의 SQL문 안에 포함된 또 다른 SQL문
- 복잡한 데이터를 이용하여 검색하거나 값을 비교할 때 사용
- 괄호로 감싸서 작성
- 사용 가능한 위치
  - SELECT 절: 조회할 컬럼이나 계산식에 서브쿼리 작성 가능
  - FROM 절: 테이블 대신 서브쿼리 작성 가능
  - WHERE 절: 검색 조건에 서브쿼리 작성 가능
  - 그 외 HAVING, INSERT문의 VALUES 절, UPDATE문의 SET 절 등에서도 사용 가능
### Single-row subquery
- 실행 결과가 항상 1건 이하인 서브쿼리
- 비교 연산자와 함께 사용
- `=`, `>`, `<`, `>=`, `<=`, `<>` 등과 함께 사용하여 서브쿼리 결과와 비교
- 예: 북아메리카 대륙의 평균 인구보다 인구가 많은 국가 조회
```SQL
SELECT country_name, population
FROM countries
WHERE population > (
    SELECT AVG(population)
    FROM countries
    WHERE continent = 'North America'
);
```
- 서브쿼리 select절에서 사용 시 가독성을 위해 별칭 사용 권장
- 단일 행 서브쿼리는 항상 단일 값 반환. where절이나 select절에서 사용하는 것이 일반덕
- 비교 연산자 사용 시 서브쿼리가 단일 행 반환해야 함. 여러 행 반환 시 오류 발생
### Multi-row subquery
- 실행 결과가 여러 개의 행을 반환할 수 있는 서브쿼리
- IN, ANY, ALL 연산자와 함께 사용. 비교 연산자 사용 시 오류
- join으로도 표현 가능, 하지만 테이블 전체의 정보까진 필요없고 특정 컬럼의 특정 조건에 맞는 값만 필요할 때 서브쿼리로 표현하는 것이 더 간결할 수 있음
- 예: 아시아 대륙의 국가의 도시들의 정보 조회
```SQL
SELECT city_name, population
FROM cities
WHERE country_id IN (
    SELECT country_id
    FROM countries
    WHERE continent = 'Asia'
);
```
### Multi-column subquery
- 실행 결과가 하나 이상의 컬럼을 반환하는 서브쿼리
- 비교 연산자나 EXISTS, IN 연산자와 함께 사용
- 복잡한 조건이 필요하면 서브쿼리가 따라오게 된다.
## Database Index
- 데이터베이스에서 데이터를 보다 빠르게 찾기 위해 사용되는 자료구조
### INDEX
- 장점: 조회 속도 빠름, 시스템 부하 적음
- 단점: 인덱스 저장을 위한 공간 필요, 삽입/수정/삭제 많을 시 성능 저하
### 종류
- 기본 인덱스
  - 일반적으로 선언한 인덱스
  - 인덱스로 설정된 컬럼의 데이터에 null 존재 가능
  - 중복 가능
- 유니크 인덱스
  - unique 키워드로 선언한 인덱스
  - 고유 값 필요. 중복시 dulpicate entry 오류 발생
### 인덱스 생성
- 테이블 생성 시 인덱스 생성
```SQL
CREATE TABLE table_name (
    column1 datatype,
    column2 datatype,
    ...
    INDEX index_name (column_name)
);
```
- **자주 검색**되고 중복되는 데이터가 적은 컬럼을 인덱스로 지정
- 인덱스 추가
```SQL
CREATE INDEX index_name ON table_name (column_name);
```
- 테이블 생성 시 인덱스 정의가 인덱스 추가보다 효율적
  - 데이터가 없기 때문에 인덱스 설정 시간이 적음
  - 테이블에 데이터가 많을 때 인덱스 추가하면 인덱스 설정을 위해 일시적으로 데이터베이스 시스템에 영향 가능. 
- 인덱스 삭제
```SQL
ALTER TABLE table_name
DROP INDEX index_name;
```
- 다른 데이터베이스 객체와 일관된 구문. 주로 사용
```SQL
DROP INDEX index_name ON table_name;
```
- 작업의 명확성 높일 때 사용
- 인덱스는 유일한 값 가진 컬럼 사용할수록 유리

## VIEW
- 데이터베이스 내에서 저장된 쿼리의 결과를 가상으로 나타내는 객체, 가상 테이블
- 실제 데이터 x, 실행 시점에 쿼리 실행하여 결과 생성
- 쿼리로 함수를 만드는 거랑은 좀 다름. input에 따라 output이 달라지는 게 아니라. vue에서 computed와 비슷할듯.
### 장점
- 편리
 - 복잡한 쿼리를 미리 정의된 뷰로 만들어 단순화 가능
- 보안성
  - 사용자가 데이터베이스 일부만 복 수 있도록 제한 가능
- 재사용성
 - 동일한 뷰를 여러 쿼리에서 사용 가능
### 생성
```SQL
CREATE VIEW view_name AS
SELECT column1, column2, ...
FROM table_name
WHERE condition;
```
- 뷰 이름은 v_같은 접두사를 이용하여 구분하는 것 권장
- as 이후에 select 구문 작성
- view_name로 뷰 접근
### 삭제
```SQL
DROP VIEW view_name;
```
## B-tree
- 데이터베이스에서 가장 일반적으로 사용하는 인덱스 구조
- B+tree와 B-tree는 다름
- 이진 탐색 트리의 한계 보완한 자료구조
- 하나의 레벨에 많은 자료를 저장할 수 있어 높이가 낮음
- 항상 O(log n) 시간 복잡도로 검색, 삽입, 삭제 가능
