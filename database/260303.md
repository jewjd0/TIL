# Introduction of Database
## Database
### 개요
- 데이터베이스: 체계적인 데이터 모음
- 데이터: 저장이나 처리에 효율적인 형태로 변환된 정보
### Relational Database
- 데이터베이스 역할: 데이터 저장, 조작
- 관계형 데이터베이스: 데이터 간에 관계가 있는 데이터 항목들의 모음
  - 테이블 행, 열의 정보 구조화하는 방식
  - 서로 관련된 데이터 포인터 저장, 이에 대한 액세스 제공
- 관계: 여러 테이블 간 논리적 연결
- 관계로 할 수 있는 것
  - 관계 있는 두 테이블 사용하여 데이터를 다양한 형식으로 조회 가능
- 키워드
  - Table(Relation): 데이터를 기록하는 곳
  - Field(Column, Attribute): 테이블의 열, 고유한 데이터 형식(타입 지정)
  - Record(Row, Tuple): 테이블의 행, 구체적 데이터 값 저장
  - Database(Schema): 테이블의 집합
  - Primary Key: 각 레코드의 고유값, 레코드의 식별자로 사용
  - Foreign Key: 다른 테이블의 레코드 식별할 수 있는 키, 다른 테이블의 Primary Key 참조, 테이블 간 관계 설정
### 정리
- Table: 데이터가 기록되는 곳
- Table에는 행에서 고유 식별 가능한 기본 키 속성 존재, 
    외래 키 사용하여 각 행에서 서로 다른 테이블 간 관계 설정 가능
- 데이터는 기본 키, 외래 키를 통해 결합(join)될 수 있는 여러 테이블에 걸쳐 구조화 됨

## RDBMS
### DBMS(Database Management System)
- 데이터베이스를 관리하는 소프트웨어 프로그램
- 데이터 저장 및 관리 용이하게 하는 시스템
- 데이터베이스- 사용자 간 인터페이스
- 사용자가 데이터 구성, 업데이트, 모니터링, 백업, 복구 등을 할 수 있도록 지원
### RDBMS(Relational Database Management System)
- 관계형 데이터베이스를 관리하는 소프트웨어 프로그램
- SQLite, MySQL, PostgreSQL, Oracle Database 등

## Database Modeling
### Database Modeling
- 데이터베이스 시스템을 구축하기 위한 과정. 데이터의 구조, 관계, 트상 결정
- 효율성, 일관성, 무결성 보장하기 위해 진행하는 단계
### 무결성 제약 조건
- 데이터베이스의 데이터 일관성 유지. 부적절한 데이터의 삽입, 수정, 삭제 방지 -> 데이터의 신뢰성 보장
1. 개체 무결성(Entity Integrity): 기본 키의 고유성 보장, NULL 값 허용 안 함
2. 참조 무결성(Referential Integrity): 외래 키와 기본 키 간의 관계 유지, 무효한 참조 방지(존재하지 않는 기본 키 참조 불가)
3. 도메인 무결성(Domain Integrity): 각 속성의 값이 정의된 도메인에 속하는 것을 보장. 속성이 가질 수 있는 값의 범위, 형식, 제약 조건 정의. 속하지 않는 경우 삽입/수정 제한, 오류 발생
4. 고유성 무결성(Unique Integrity): 특정 컬럼의 값이 테이블 내에서 중복되지 않도록 제한
5. NULL 무결성(NULL Integrity): 특정 컬럼이 NULL 값을 가질 수 없도록 강제
6. 일반 무결성(General Integrity): 비즈니스 로직에 따라 추가 정의하는 무결성 규칙. 예: 은행 계좌 잔액이 0 미만으로 내려가지 않음

## Proceeding with Database Modeling
### 데이터베이스 모델링
- 요구사항 수집/분석 -> 개념적 설계 -> 논리적 설계 -> 물리적 설계
### 요구사항 수집 및 분석
- 어떤 종류의 데이터를 정리하는지 정보 수집, 어떤 작업 수행하는지 파악
- 개체(Entity): 업무에 필요하고 유용한 정보를 저장하는 집합적인 것
- 속성(Attribute): 관리하고자 하는 것의 의미를 더 이상 작은 단위로 분리되지 않은 데이터 단위
- 관계(Relationship): 객체 사이의 논리적인 연관성
### 개념적 설계
- 요구사항 기반으로 데이터베이스의 개념적 모델 설계
- 객체/관계 식별, 객체 간 관계 정의히여 ER Diagram 작성.
- 까마귀 발 모델 표기법 사용
- n:m 관계는 중간 테이블로 표현하여 1:n 관계로 변환
  - n:m 관계는 foreign key로 표현 할 수 없기 때문.
  - erd 그릴 때는 내가 혹은 팀끼리 알아볼 수 있는 형태로 만들 것. 
### 논리적 설계
- 개념적 설계 기반으로 데이터베이스 논리적 구조 설계
- 테이블, 칼럼(속성), 제약 조건 등 구체적 데이터베이스 개체 정의
- 정규화 수행, 데이터 중복 최소화 및 일관성 유지
### 물리적 설계
- 논리적 설계 기반으로 데이터베이스 실제 저장/운영 가능한 형태로 변환
- 테이블의 인덱스, 파티션, 클러스터링 등 물리적 구조/접근 방식 결정
- 보안, 백업/복구, 성능 최적화 등 고려하여 데이터베이스 설정

# Database Normalization
### Database Normalization
- 정규화(Normalization): 중복을 최소화하고 데이터의 일관성과 효율성 유지를 위해 데이터를 구조화하는 과정
- 정규화의 목적: 중복 최소화, 이상 현상 방지, 데이터 구조 변경 시 다시 구조화 하는 영역 최소화
- 이상 현상(Anomaly): 데이터베이스를 잘못 설계 했을 때 발생 가능한 불필요한 데이터 중복으로 인한 부작용. 
  - 삽입 이상: 새 데이터를 삽입하기 위해 불필요한 데이터 삽입 필요한 경우
  - 갱신 이상: 중복된 데이터 중 일부만 변경 -> 데이터 불일치 발생
  - 삭제 이상: 데이터 삭제 시 반드시 필요한 다른 데이터가 함께 삭제되는 경우

## 정규형(Normal Form)
- 데이터베이스 정규화 종류: 1NF ~ 6NF (+ BCNF) 가 있지만, 일반적으로 1NF ~ 3NF 까지 적용하는 경우가 많음
### 제 1 정규화(1NF)
- 각 속성이 원자적이어야 함
  - 각 속성이 원자값(atomic value, 하나의 값)만을 가져야 함
- 각 행이 유일한 식별자(Primary Key)를 가져야 함
  - 중복 데이터 허용 안함
- 특정 고객의 전화번호가 여러 개일 때, 전화번호를 별도의 테이블로 분리, 고객 테이블을 참조하게 하는 방식으로 정규화
- 한 테이블에 여러 값이 있는 경우 제 1 정규화 고려
### 제 2 정규화(2NF)
- 제 1 정규화 만족 + 부분적 함수 종속 제거
  - 테이블의 모든 컬럼이 기본키에 대해 완전히 종속
  - 기본 키의 일부만으로 결정되는 속성 제거(새 테이블로 분리)
  - 부분 함수 종속 제거
  - 상품명과 생산자가 상품 가격에 종속적, 생산자는 생산지에만 종속적일 때 생산자/생산지, 생산자/상품명/상품 가격으로 테이블 분리
  - 복합키가 있는 경우에 복합키 모두에 종속되지 않는 속성이 있으면 제 2 정규화 고려
### 제 3 정규화(3NF)
- 제 2 정규화 만족 + 이행적 함수 종속(Transitive Dependency) 제거 
- 이행적 함수 종속
  - A->b, B->C 일 때 A->C인 경우
  - 기본 키가 아닌 다른 속성에 종속된 경우
- 기본 키가 아닌 속성 간의 종속성 제거
- 학생 테이블에서 학생번호 -> 학과, 학과 -> 학과장인 경우
  - 비 기본 키 컬럼이 다른 비 기본 키 컬럼을 결정하고 있음.
  - 학생 정보와 학과 정보 분리 -> 학과/학과장, 학생번호/학생 이름/학과로 테이블 분리
  - 기본 키가 아닌 것에 종속된 것이 있을 경우 제 3 정규화 고려
### BCNF(Boyce-Codd Normal Form)
- 후보 키(Candidate Key)
  - 테이블에서 각 행을 유일하게 식별할 수 있는 속성 중 최소성 만족하는 키
  - pk의 후보가 될 수 있어서 후보 키라고 불리는 것
  - 유일성, 최소성 만족
- 제 3 정규화 만족 + 모든 결정자가 후보 키인 경우
  - 결정자: 다른 속성을 결정하는 속성
  - 제 3 정규형을 적용한 후에도 여전히 이상현상 발생할 수 있음
  - 3NF는 후보 키가 아닌 속성이 결정자가 되는 문제를 해결하지 못할 수 있음
- 모든 결정자가 후보 키가 되도록 강제 -> 더 엄격한 정규화
  - 기본 키가 아닌 속성이 다른 속성을 결정할 수 없어야 함
- 담당 교수가 하나의 과목만 맡고 있을 때 학생번호/과목명/담당교수/학점 이 있는 테이블에서 담당 교수가 과목명을 변경할 가능성 존재
 - 담당 교수와 과목명을 과목 테이블로 분리 -> 과목명/담당 교수, 학생번호/과목명/학점으로 테이블 분리

# SQL
### SQL(Structured Query Language)
- 데이터베이스에 정보를 저장하고 처리하기 위한 프로그래밍 언어
- 관계형 데이터베이스와의 상호 작용을 위해 사용
- 데이터 정의, 조작, 제어 기능 제공
### SQL Syntax
- `SELECT column_name FROM table_name;`
- SQL 키워드는 대소문자 구분 안함
  - 대문자 작성 권장(명시적 구분)
  - 팀원과 협의하여 일관된 스타일 유지
- SQL 문장 끝에는 세미콜론(;) 사용 필요
## SQL Statements
- SQL을 구성하는 가장 기본적인 코드 블록
- SQL 문장 종류
  - DDL(Data Definition Language): 데이터베이스 구조 정의하는 명령어
    - CREATE, ALTER, DROP 등
  - DQL(Data Query Language): 데이터 조회하는 명령어
    - SELECT 등
  - DML(Data Manipulation Language): 데이터 조작하는 명령어
    - INSERT, UPDATE, DELETE 등
  - DCL(Data Control Language): 데이터베이스 접근 제어하는 명령어
    - GRANT, REVOKE 등
  - TCL(Transaction Control Language): 트랜잭션 제어하는 명령어
    - COMMIT, ROLLBACK 등
  - DQL, DML을 합쳐서 DML로 분류하기도 함
  - DCL, TCL을 합쳐서 DCL로 분류하기도 함
