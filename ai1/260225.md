# 프롬프트 엔지니어링

## 최종 목표
- 왜 작업 분해가 필요한지 알기
- 이 역할을 코드 변경 책임과 어떻게 연결하는지 알기
- 이 역할들을 누가, 어떤 순서로 실행시키는지 알기

## 개요
### 기존 개발 방식의 한계
- 기존 개발 방식의 한계는 비교적 단순
  - 반복 작업 많음
  - 사소한 코드 작성에 시간이 많이 걸림
  - 맥락 전환 잦음
  => 해결을 위해 AI 코딩 어시스턴트 등장

### 에이전트 개념의 등장 배경
- 하나의 ai에게 여러 종류의 일을 동시에 맡기면 실패
- -> 역할을 나눠서 실행하게
- 역할들을 사람 또는 시스템이 순서대로 호출

### ai 에이전트란?
- reason: 지금 상황에서 무엇을 해야 할지 판단
- act: 선택 행동 실행
- observe: 실행 결과 확인
- 이 결과를 다시 reason으로 피드백하여 다음 행동 결정

### ai 에이전트는 만능인가?
- 추론 과정을 거쳐 검증 결과 도출 -> 실패하지 않는 ai?
- 에이전트는 여러 단계를 자동 실행 가능하지만 실패 양상도 기존과 다를 뿐 실패함
  - 변경 범위 증가
  - 실패 원인 파악 어려움
  - 사람이 검증하기 전에 다음 단계로 넘어감
### 에이전트 실패 사례
- 잘못된 연쇄 실행
  - 초기 판단이 틀려도 그 판단을 전제로 다음 행동 이어감
- 책임 위치 불명확
  - 에이전트가 여러 단계 자동 실행하면 다음 질문에 답하기 어려움
  - 왜 변경했는가?, 누가 결정을 내렸는가? 어디까지 의도된 수정인가?

## 프롬프트 엔지니어링
- ai에게 무엇을 어떻게 지시할지 설계하는 기술
  1. prompt: ai에게 주는 자연어 지시나 질문
  2. 목표: 모델이 사용자 의도와 일치하는 답을 내도록 하는 것
  3. 엔지니어링: 단순 질의 작성이 아닌 구조적 설계와 반복적 개선

### 프롬프트 엔지니어링이 필요한 이유
- 프롬프트는 단순 입력이 아닌 ai 출력을 안내하는 인터페이스
- 모델의 응답 품질은 프롬프트의 구조, 명료성, 맥락 정보에 크게 영향ㅇ받음
  - 모델이 의도를 정확히 이해하게 함
  - 불명확, 모호한 응답 방지
  - 결과의 일관성과 정확성 향상
### 좋은 프롬프트의 기본 구조
- 역할, 맥락 지정
- 명확하고 구체적 지시
- 예시 제공
- 맥락, 입력 정보 제공
- 검증 기준 명시




### 계획 단계
- 역할 선언 - 만드는 역할이 아닌 정리하는 역할
- 목효 정의 - 코드가 아니라 구조화된 계획
- 입력 명시 - 결과가 아닌 요구를 입력으로 받음
- 금지사항 - 계획 단계는 판단과 구조화까지
- 출력 구조 - 다음 단계가 검증할 수 있어야
