## 클라이언트와 서버

- 클라이언트 - 서버 구조

- 서버 : 요청 응답 주체
## Frontend와 Backend
- Frontend(프론트엔드)
  - 사용자 인터페이스 구성, 사용자가 애플리케이션과 상호작용할 수 있도록 함
  - -> HTML, CSS, JavaScript, 프론트엔드 프레임워크 등
- Backend(백엔드)
  - 서버 측에서 동작, 클라이언트의 요청에 대한 처리와 데이터베이스와의 상호작용 등 담당
  - -> 서버 언어(예: Python, Java 등) 및 백엔드 프레임워크, 데이터베이스, API, 보안 등

# Django Framework
## 개요
- Django : Python 기반 대표적 웹 프레임워크
### 왜 Django를 사용할까?
- 다양성: 파이썬 기반으로 소셜 미디어 및 빅데이터 관리 등 광범위한 서비스 개발에 적합
- 확장성: 대량의 데이터에 대해 빠르고 유연하게 확장할 수 있는 기능 제공
- 보안: 취약점으로부터 보호하는 보안 기능이 기본적으로 내장
- 커뮤니티 지원: 개발자를 위한 지원, 문서 및 업데이트 제공하는 활성화 된 커뮤니티
- 검증된 웹 프레임워크: 대규모 서비스에서도 안정적인 서비스 제공 (예: Instagram, Dropbox 등)
- 기업에서는 spring기반 자체 프레임워크 많이 사용 <br>
  => 결국 새로 배워야 하는데 어떤 프레임워크를 배우냐 보다는 CRUD를 잘 이해하는 게 중요
## 가상환경
- python 애플리케이션과 그에 따른 패키지들을 격리하여 관리할 수 있는 **독립적인** 실행 환경
### 언제 필요한가?
- 프로젝트 A, B가 같은 패키지의 각각 다른 버전을 필요로 할 경우
- 프로젝트마다 필요한 서로 다른 패키지가 충돌이 발생할 경우
### 1. 가상환경 생성
- `python -m venv <가상환경 이름>` 
  - 예시: `python -m venv venv`
  - 대체로 `venv`, `.venv` 라는 이름으로 가상환경 생성
### 2. 가상환경 활성화
- `source venv/Scripts/activate`
- 가상환경 활성화 시 프롬프트에 가상환경 이름이 표시
- acitivate하면 가상환경 내에 설치된 패키지, 라이브러리 사용하게 
### 3. 환경에 설치된 패키지 목록 확인
- `pip list`
- 패키지 목록 필요한 이유
  - clone 받고 실행할 때 정보를 알아야 한다.
  - 프레임워크 업데이트 시 돌아가지 않는 경우 있어 구버전 필요할 수 있음
### 4. 가상환경 비활성화
- `deactivate`

### 4. 장고 프로젝트 생성 및 시작
- `django-admin startproject <프로젝트 이름> .`
  - 예시: `django-admin startproject firstpjt .`
  - 프로젝트 디렉토리 생성, 기본 파일과 폴더 구조 자동으로 생성
  - 마지막 `.`는 현재 디렉토리에 프로젝트 생성하라는 의미
- `python manage.py runserver`
  - 개발 서버 실행, 웹 브라우저에서 개인 서버 주소로 접속하여 확인 가능
### 의존성 패키지
- `pip freeze > requirements.txt`
  - 현재 가상환경에 설치된 패키지와 버전 정보를 `requirements.txt` 파일로 저장
  - 프로젝트를 다른 환경에서 실행할 때 필요한 패키지와 버전 정보를 제공
  - `pip install -r requirements.txt` 명령어로 `requirements.txt` 파일에 명시된 패키지와 버전을 설치하여 동일한 환경 구성 가능
  - 이렇게 습관화하기! 프로젝트에 필요한 패키지와 버전 정보를 명확히 관리하는 것이 중요
- 가상환경, 의존성 패키지 관리!
## Django Design Pattern
### 개요
- 디자인 패턴 : 소프트웨어 설계에서 발생하는 문제를 해결하기 위한 일반적인 해결책
- 공통적인 문제를 해결하는 데 쓰이는 형식화 된 관행
- 애플리케이션의 구조는 이렇게 구성하자는 관행
### MVC(Model-View-Controller) 패턴
- 애플리케이션을 구조화하는 대표적 패턴
- 데이터, 사용자 인터페이스, 비즈니스 로직 분리
- => 시각적 요소, 뒤에서 실행되는 로직을 서로 영향 없이, 독립적이고 쉽게 유지보수 가능한 애플리케이션 구축 위해
- Model : 데이터와 비즈니스 로직 담당
- View : 사용자 인터페이스와 사용자 경험 담당   
- Controller : 사용자 입력 처리, 모델과 뷰 간의 상호작용 담당
### MVT(Model-View-Template) 패턴
- Django에서 사용하는 디자인 패턴
- MVC 패턴과 유사하지만, Django에서는 Controller 대신 Template 사용
- 단순히 명칭만 다르게 정의한 것
- Model : 데이터와 비즈니스 로직 담당
- Template : 사용자 인터페이스의 시각적 요소 담당, HTML과 Django 템플릿 언어 사용하여 작성
  - 수업에서는 자세히 다루지 않을 것. api만드는 데 집중할거기 때문
- View : 사용자 인터페이스와 사용자 경험 담당
## Django Project와 App
- Django 프로젝트 : 애플리케이션의 집합. db설정, url 연결, 전체 앱 설정 등 처리
- 앱과 같은 폴더에 있지만 프로젝트 내부에 프로젝트를 관리하기 위한 폴더를 startproject로 생성하는 것
- 같은 폴더에 프로젝트 여러 개 둘 수 있음
- vue에서 app하나 있고 component여러 개 있는 거랑 비슷. 같지는 않으니 유의. js에서 ==는 true, ===는 false 정도의 느낌
### 프로젝트 폴더
- manage.py : 프로젝트 관리 명령어를 실행하기 위한 파일
- settings.py : 프로젝트 설정 파일, 데이터베이스 설정, 앱 등록, 보안 설정 등 전반적인 설정 관리
- urls.py : 요청 들어오는 url에 따라 해당하는 적절한 view로 연결
  - controller 역할은 아님. 앱 같은 기능들이 
- __init__.py : 해당 디렉토리가 파이썬 패키지임을 나타내는 파일(빈 파일) (요즘버전은 없어도 되는데 호환성 문제로 있는 게 좋음)
- asgi.py : 비동기식 웹 서버와의 연결 관련 설정
- wsgi.py : 웹 서버와의 연결 관련 설정
### 앱 구조
- admins.py : 관리자 사이트 설정 파일, Django의 관리자 인터페이스에 모델 등록 및 커스터마이징
- models.py : 데이터 모델 정의 파일, 데이터베이스 테이블과 상응하는 모델 클래스 정의(내일다룸)
- views.py : http 요청 처리, 해당 요청에 대한 응답 반환
  - mtv 패턴의 v
- apps.py : 앱 설정 파일, 앱의 이름과 구성을 정의
- tests.py : 앱의 테스트 케이스 작성 파일, 앱의 기능이 올바르게 작동하는지 확인하기 위한 테스트 코드 작성
### 앱 생성
- `python manage.py startapp <앱 이름>`
  - 예시: `python manage.py startapp articles`
  - 앱 디렉토리 생성, 기본 파일과 폴더 구조 자동으로 생성
  - 따로 경로지정은 안해도 됨
  - 앱 이름 복수형 지정 권장
  - `__init__.py` : 해당 디렉토리가 파이썬 패키지임을 나타내는 파일(빈 파일) (요즘버전은 없어도 되는데 호환성 문제로 있는 게 좋음)
### 앱 등록
- 프로젝트의 `settings.py` 파일에서 `INSTALLED_APPS` 리스트에 앱 이름 추가하여 등록
  - 예시: `INSTALLED_APPS = [ ..., 'articles', ]`
  - 앱 등록은 프로젝트가 앱을 인식하고 사용할 수 있도록 하는 과정
  - 반드시 앱을 생성한 후 등록할 것! 등록 후 생성은 불가.

# REST API
## 개요
### API(Application Programming Interface)
- 두 소프트웨어가 서로 통신할 수 있게 하는 메커니즘
- 클라이언트-서버처럼 서로 다른 프로그램에서 요청과 응답을 받을 수 있게 만든 체계
### Web API
- 웹 서버, 웹 브라우저를 위한 API
- 현대 웹 개발은 여러 open api들을 활용하는 추세
- 예시: 구글 맵 API, 유튜브 API, 네이버 파파고 API 등
### REST API(Representational State Transfer API)
- api server를 개발하기 위한 일종의 소프트웨어 설계 방법론
- 모두가 설계하는 구조가 다르니 맞춰서 설계하자는 합의
- 규칙은 아님.
### RESTful API
- REST API의 원칙을 준수하는 API
- **자원을 정의**하고 **자원에 대한 주소를 지정**하는 전반적인 방법 서술
### REST API
- REST라는 설계 디자인 약속을 지켜 구현한 API
- 자원의 식별, 행위, 요청으로 구분
## 자원의 식별
### URI(Uniform Resource Identifier)
- 인터넷에서 리소스(자원)을 식별하는 문자열
- 가장 일반적인 형태는 URL(Uniform Resource Locator)
- https로 만드는 방법을 알고 나서 배포할 것. 요즘 무료로 바뀜
- Schema(or Protocol) 
  - 브라우저가 리소스를 요청하는 데 사용해야 하는 규약
  - url의 첫부분은 브라우저가 어떤 규약을 사용하는지 나타냄
  - https , notions 등
- Domain Name
  - 요청 중인 웹 서버
  - 직접 ip주소 입력해도 되지만, 도메인 네임 시스템(DNS)을 통해 사람이 읽을 수 있는 이름으로 변환하여 사용
- Port
  - 웹서버의 리소스에 접근하는 데 사용되는 기술적인 문
  - 표준 포트만 작성 시 생략 가능
  - http: 80, https: 443
- Path
  - 웹 서버의 리소스 경로
  - 초기에는 실제 파일 위치, 지금은 추상화된 형태의 구조 표현
- Parameters
  - &로 구분되는 key-value 쌍으로, 서버에 추가 정보 전달
- Anchor
  - 일종의 북마크, 브라우저에 해당 지점에 있는 콘텐츠 표시
## 자원의 행위
### HTTP Request Method
- 리소스에 대한 행위를 정의 
- http verbs라고도 함
- GET : 서버에 리소스 표현 요청, get을 사용하는 요청은 데이터만 검색
- POST : 데이터를 지정된 리소스에 제출, 서버 상태 변경
- PUT : 요청한 주소의 리소스 수정
- DELETE : 지정된 리소스 삭제
- 프론트엔드에서 axios로 요청할 때 경로는 별도로 작성했음. 그렇게 구분지어서 작동하도록 만들 것.
- 프론트에서 사용할 땐 get, post만 쓰는 게 좋다는 것 같은데 만들 땐 patch까지도 만들거임
## 요청과 응답
### HTTP response status code
- 클라이언트의 요청에 대한 서버의 응답 상태를 나타내는 3자리 숫자 코드
- informational (1xx) : 요청이 수신되어 처리 중임을 나타냄
- successful (2xx) : 요청이 성공적으로 처리되었음을 나타냄 
  - 200 OK : 요청이 성공적으로 처리되었음을 나타냄
  - 201 Created : 요청이 성공적으로 처리되어 새로운 리소스가 생성되었음을 나타냄
  - 204 No Content : 요청이 성공적으로 처리되었지만, 반환할 콘텐츠가 없음을 나타냄
- redirection (3xx) : 클라이언트가 요청한 리소스가 다른 위치로 이동되었음을 나타냄
  - 전화돌리기 
- client error (4xx) : 클라이언트의 요청에 오류가 있음을 나타냄
  - 400 Bad Request : 클라이언트의 요청이 잘못되어 서버가 이해할 수 없음을 나타냄
  - 401 Unauthorized : 인증이 필요한 리소스에 대한 요청이 인증되지 않았음을 나타냄
  - 403 Forbidden : 클라이언트가 요청한 리소스에 접근할 권한이 없음을 나타냄
  - 404 Not Found : 요청한 리소스를 찾을 수 없음을 나타냄
- server error (5xx) : 서버가 요청을 처리하는 중에 오류가 발생했음을 나타냄
- API명세서에 명시해주는 게 좋다!
- 문서 작성해주는 프레임워크 있으니 활용할 것
### DRF(Django REST Framework)
- Django에서 REST API를 쉽게 구축할 수 있도록 도와주는 라이브러리
- `pip install djangorestframework` 명령어로 설치
- 프로젝트 settings.py 파일의 INSTALLED_APPS 리스트에 'rest_framework' 추가하여 등록
- 의존성 파일 생성 `pip freeze > requirements.txt`
- install명령어와 setting에 등록하는 단어는 왜 다른가? python 컨벤션 때문에 snake_case 작성, pip에 등록한 패키지 이름이 djangorestframework이기 때문

## 참고
### render 함수
- 주어진 템플릿을 주어진 컨텍스트 데이터와 결합, 렌더링 된 텍스트와 함께 httpResponse 응답 객체를 반환하는 함수
- request: 응답 생성에 사용되는 요청 객체
- template_name: 렌더링할 템플릿의 이름
- context: 템플릿에서 사용할 데이터(딕셔너리 타입)
- `render(request, template_name, context)`
